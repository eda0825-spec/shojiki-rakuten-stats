# Lark Base スキーマ設計 (SHOJIKI 工場連携 + VOC)

中国工場と日本エンジニアの不具合共有 + Amazon/楽天レビューの集約に使う Lark Base
3 ベースの列定義。ユーザーがこのまま Lark で作成できる形にしてある。

> 中国側は飞书 (Lark の中国版)、日本側は Lark で同じワークスペースに参加できる。
> Google/Notion/Slack が遮断される中国でも素で動く。

---

## Base 1: `SH-J001 工場連携`

メンバー: J001 を作る中国工場 + 日本エンジニアチーム
URL 短縮名: `shj001-defects`

### テーブル: `不具合・改善要望`

| 列名 (日本語) | 列名 (中国語) | 型 | オプション / 備考 |
|---|---|---|---|
| ID | 编号 | 自動採番 (J001-0001 形式) | 主キー |
| 発生日 | 发现日期 | 日付 | デフォルト: 今日 |
| 報告者 | 报告人 | メンバー | 自動: 作成者 |
| 報告元 | 来源 | 単一選択 | 工場検査 / 顧客レビュー / サポート / 社内テスト |
| ロット番号 | 批次号 | テキスト | 例: `2026-W18` |
| シリアル番号 | 序列号 | テキスト | 個体追跡用、任意 |
| 工程 | 工序 | 単一選択 | 組立 / 検査 / 包装 / 部品 / 出荷後 |
| 部位 | 部位 | 複数選択 | バッテリー / モーター / ローラー / ダストカップ / フィルター / 充電端子 / スタンド / 操作ボタン / その他 |
| 症状 (JP) | 症状 (日) | 長文 | 日本語の症状記述 |
| 症状 (ZH) | 症状 (中) | 長文 | 中国語の症状記述 (相互翻訳) |
| 写真 | 照片 | 添付ファイル | 複数可 |
| 動画 | 视频 | 添付ファイル | Lark 上で再生可、20GB/file まで |
| 深刻度 | 严重度 | 単一選択 | 高 (安全性) / 中 (機能停止) / 低 (軽微) |
| 発生頻度 | 发生频率 | 単一選択 | 単発 / 散発 / 多発 / 全数 |
| 想定原因 | 推测原因 | 長文 | 工場側の所見 |
| 対応案 | 对策 | 長文 | エンジニア側の対策案 |
| ステータス | 状态 | 単一選択 | 未対応 / 調査中 / 対策実施中 / 再発防止 / 完了 / 様子見 |
| 担当 (工場) | 工厂负责人 | メンバー |  |
| 担当 (日本) | 日本负责人 | メンバー |  |
| 期限 | 截止日 | 日付 |  |
| 関連レビュー | 关联评论 | リンク (Base 3 `VOC` への参照) | 楽天/Amazon レビュー由来の不具合と紐付け |
| 更新日 | 更新日 | 自動: 最終更新日時 |  |

### 自動化ルール (Lark Automation)

1. 新規レコード作成時 → 担当(日本) と 担当(工場) に通知
2. ステータス変更時 → 報告者と担当全員に通知
3. 深刻度=高 → グループチャットへ即時通知 (`#shojiki-quality-alert`)
4. ステータス=完了 後 7日経過しても変更無し → 自動アーカイブ

### ビュー

- 🔴 **未対応 (深刻度順)**: ステータス=未対応 OR 調査中、深刻度降順
- 📋 **進行中 (担当別)**: 担当(工場) でグループ
- 🗂 **完了アーカイブ**: ステータス=完了
- 📊 **部位別集計**: 部位でグループ、件数表示

## Base 2: `SH-J002 工場連携`

メンバー: J002 を作る中国工場 + 日本エンジニアチーム
URL 短縮名: `shj002-defects`

**Base 1 と同じスキーマ**。コピーで作成して、ID プレフィックスだけ `J002-0001` に変える。

> ⚠ J001 と J002 は別工場・別シリーズ。Base を分けることで工場間の情報漏洩を防ぐ。
> 日本エンジニアは両方の Base に参加するが、各工場メンバーは自社の Base にしか入れない。

## Base 3: `カスタマー VOC` (Voice of the Customer)

メンバー: 日本エンジニア + マーケ。工場メンバーは入れない。
URL 短縮名: `shojiki-voc`

楽天/Amazon の生レビューを自動取り込みし、改善要望や不具合を抽出して
工場連携 Base へ転記するための中継テーブル。

### テーブル: `レビュー`

| 列名 | 型 | オプション / 備考 |
|---|---|---|
| ID | テキスト | 取得元の固有ID (例: `rkt-shj001-bd2a107d`) |
| 取得日 | 日付 | バッチ実行日 |
| モール | 単一選択 | 楽天 / Amazon / Judge.me / その他 |
| 商品 | 単一選択 | SH-J001 / SH-J002 |
| 星 | 数値 (1-5) |  |
| レビュー日 | 日付 | レビュー投稿日 |
| 投稿者 | テキスト | nickname |
| 本文 (JP) | 長文 | 原文 |
| 本文 (ZH) | 長文 | Claude による中国語訳 |
| 要約 (JP) | テキスト | Claude による 30-60字要約 |
| 要約 (ZH) | テキスト | Claude による中国語要約 |
| 分類 | 単一選択 | 不具合 / 改善要望 / 称賛 / 質問 / その他 |
| 深刻度 | 単一選択 | 高 / 中 / 低 / 該当なし |
| トピック | 複数選択 | バッテリー / 充電 / 重量 / 吸引力 / 操作性 / 音 / etc |
| 対策案 | 長文 | Claude が提案する action_hint |
| 工場転記済み | チェックボックス | 該当 Base に Issue を作ったらON |
| 関連 Issue | リンク | Base 1 or Base 2 のレコードへ |
| URL | URL | 元レビューのリンク |

### 自動化ルール

1. 分類=不具合 で深刻度=高 → Slack/Lark 通知
2. 工場転記済み=false かつ 分類=(不具合 OR 改善要望) で7日経過 → リマインダ通知

### ビュー

- 🚨 **対応必要**: 分類=(不具合 OR 改善要望) AND 工場転記済み=false
- 📈 **トピック別集計**: トピックでグループ、件数降順
- 🟢 **称賛**: 分類=称賛 (LP / SNS 用のネタ探し)
- 🆕 **新着 (直近30日)**: レビュー日 ≥ 30日前

## 列定義のコピーを楽にする手順

1. Lark Base を新規作成 → `スプレッドシートからインポート` 選択
2. このリポジトリの `docs/lark-schema/` 配下の CSV (後で生成) をインポート
3. インポート後、各列の型を手動で設定 (Lark のインポートは型推測が弱いため)

## API 連携 (将来)

`categorized-sh-j001.json` / `categorized-sh-j002.json` を読んで、
Lark Open API (`bitable.v1.appTableRecord.batchCreate`) で Base 3 にレコード POST する。

必要な認証情報 (準備でき次第 GitHub Secrets に追加):

- `LARK_APP_ID` (Lark Developer Console で作成したアプリ)
- `LARK_APP_SECRET`
- `LARK_VOC_APP_TOKEN` (Base 3 の `?appToken=` から取得)
- `LARK_VOC_TABLE_ID` (テーブルの `?tableId=`)

スクリプト雛形: `lark-push-voc.py` (リポジトリルート、credential 投入待ち)
